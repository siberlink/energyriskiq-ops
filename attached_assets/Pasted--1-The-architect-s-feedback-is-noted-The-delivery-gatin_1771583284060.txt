9.	The Trigger
10.	Delivery is triggered by calling one of two internal API endpoints:
•	POST /internal/run/pro-delivery
•	POST /internal/run/geri-delivery (same thing, kept for backward compatibility)
11.	Both require a runner token in the header for security. This means only authorized systems can trigger a send — not random visitors.
12.	Right now, there is no automatic scheduler built in. Someone or something has to call that endpoint. That could be:
•	You manually from the admin dashboard or a tool like Postman
•	An external cron service (e.g., cron-job.org, GitHub Actions, or a scheduled deployment) that hits the endpoint once daily
________________________________________
13.	How Emails Are Actually Sent
1.	The delivery worker calls send_email_v2() in the channel adapters
2.	That checks: Is email enabled? Is Brevo configured? Rate limit OK?
3.	If all checks pass, it calls the underlying send_email() which sends via the Brevo API (your email service)
4.	Returns success/failure, which gets recorded in the database
________________________________________
14.	How Telegram Messages Are Actually Sent
1.	The delivery worker calls send_telegram_v2() in the channel adapters
2.	That checks: Does the user have a chat ID? Is the Telegram bot token configured?
3.	If all checks pass, it calls send_telegram() which sends via the Telegram Bot API using your bot token
4.	Returns success/failure, recorded in the database
________________________________________
15.	Safety Mechanisms
•	Advisory lock prevents two delivery runs from happening simultaneously
•	Duplicate check — if a user already received a delivery today, they're skipped
•	Rate limiting on both channels
•	Runner token required to trigger — can't be called without authorization
