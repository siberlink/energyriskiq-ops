# EnergyRiskIQ Blog — Educational Content Platform

## Overview

The EnergyRiskIQ Blog is a public educational content platform at `/blog` where visitors can read articles on energy risk, geopolitics, and market intelligence. It features a separate user system from the main EnergyRiskIQ platform — blog users can register, sign in, write articles, and comment. Visitors can also comment as guests without registering. All user-submitted articles go through an admin approval workflow before being published.

## Key Design Decisions

- **Separate User System:** Blog users (`blog_users` table) are completely independent from EnergyRiskIQ platform users (`users` table). This allows public participation without requiring a paid subscription.
- **Cookie-Based Auth:** Blog authentication uses `blog_token` cookies with database-persisted sessions (`blog_sessions` table), not the `X-User-Token` header system used by the main platform.
- **Server-Rendered Pages:** All blog pages are server-rendered HTML (not SPA), generated by FastAPI route handlers. This enables SEO indexing and fast initial page loads.
- **Admin Integration:** Blog management is embedded in the existing `/admin` dashboard, reusing the `verify_admin_token` authentication system.

## Architecture

```
Visitor → /blog (public listing)
       → /blog/{slug} (article + comments)

Blog User → /blog/write (submit article → pending)
         → /blog/my-posts (track submissions)

Admin → /admin Blog section
     → Create + publish articles directly
     → Approve/reject user submissions
     → Delete any post
```

### Article Status Workflow

```
User submits article → PENDING → Admin approves → PUBLISHED
                              → Admin rejects  → REJECTED (with reason)

Admin creates article → PUBLISHED (immediately)

Admin can also:
  PUBLISHED → DRAFT (unpublish)
  REJECTED  → PUBLISHED (reconsider)
  DRAFT     → PUBLISHED (republish)
```

## Database Schema

### `blog_users` table

```sql
CREATE TABLE blog_users (
    id SERIAL PRIMARY KEY,
    display_name VARCHAR(100) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,      -- bcrypt hashed
    avatar_color VARCHAR(7) DEFAULT '#3b82f6', -- random color on creation
    bio TEXT DEFAULT '',
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### `blog_sessions` table

```sql
CREATE TABLE blog_sessions (
    token VARCHAR(64) PRIMARY KEY,           -- secrets.token_hex(32)
    user_id INTEGER NOT NULL REFERENCES blog_users(id) ON DELETE CASCADE,
    expires_at TIMESTAMP NOT NULL,           -- 30 days from creation
    created_at TIMESTAMP DEFAULT NOW()
);
```

### `blog_posts` table

```sql
CREATE TABLE blog_posts (
    id SERIAL PRIMARY KEY,
    title VARCHAR(500) NOT NULL,
    slug VARCHAR(600) NOT NULL UNIQUE,        -- auto-generated from title
    excerpt TEXT DEFAULT '',                   -- summary shown on cards
    content TEXT NOT NULL,                     -- Markdown content
    cover_image TEXT DEFAULT '',               -- optional URL
    category VARCHAR(100) DEFAULT 'General',
    tags TEXT DEFAULT '',                      -- comma-separated
    author_id INTEGER,                        -- NULL for admin-created posts
    author_name VARCHAR(100) NOT NULL DEFAULT 'Admin',
    author_type VARCHAR(20) NOT NULL DEFAULT 'admin', -- 'admin' or 'user'
    status VARCHAR(20) NOT NULL DEFAULT 'pending',    -- pending/published/rejected/draft
    rejection_reason TEXT DEFAULT '',
    view_count INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    published_at TIMESTAMP                    -- set when status becomes 'published'
);
```

### `blog_comments` table

```sql
CREATE TABLE blog_comments (
    id SERIAL PRIMARY KEY,
    post_id INTEGER NOT NULL REFERENCES blog_posts(id) ON DELETE CASCADE,
    user_id INTEGER REFERENCES blog_users(id) ON DELETE SET NULL,
    guest_name VARCHAR(100),                  -- for non-authenticated commenters
    content TEXT NOT NULL,
    is_approved BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### Indexes

```sql
CREATE INDEX idx_blog_posts_slug ON blog_posts(slug);
CREATE INDEX idx_blog_posts_status ON blog_posts(status);
CREATE INDEX idx_blog_posts_created ON blog_posts(created_at DESC);
CREATE INDEX idx_blog_comments_post ON blog_comments(post_id);
CREATE INDEX idx_blog_users_email ON blog_users(email);
```

## API Endpoints

### Public Pages (GET, return HTML)

| Route | Description |
|-------|-------------|
| `GET /blog` | Blog home page with article cards, category filters, search, pagination |
| `GET /blog/{slug}` | Individual article page with full content and comments |
| `GET /blog/write` | Write article form (requires blog auth, otherwise shows sign-in prompt) |
| `GET /blog/my-posts` | User's submitted articles with status badges (requires blog auth) |

Query parameters for `/blog`:
- `page` (int, default 1) — pagination
- `category` (string) — filter by category
- `search` (string) — search in title and excerpt

### Blog Auth API (POST, return JSON)

| Route | Description |
|-------|-------------|
| `POST /api/blog/auth/register` | Register a new blog user |
| `POST /api/blog/auth/login` | Sign in to blog account |
| `POST /api/blog/auth/logout` | Sign out and delete session |

**Register request body:**
```json
{
    "display_name": "Jane Doe",
    "email": "jane@example.com",
    "password": "securepass123"
}
```

**Login request body:**
```json
{
    "email": "jane@example.com",
    "password": "securepass123"
}
```

Validation rules:
- Display name: required, max 100 characters
- Email: required, must be unique among blog users
- Password: required, minimum 6 characters

On success, both register and login set a `blog_token` cookie (HttpOnly, Secure, SameSite=Lax, 30-day expiry).

### Blog Content API (POST, return JSON)

| Route | Description |
|-------|-------------|
| `POST /api/blog/posts` | Submit a new article (requires blog auth) |
| `POST /api/blog/posts/{id}/comments` | Add a comment to a post |

**Create post request body:**
```json
{
    "title": "Understanding Energy Risk",
    "content": "## Introduction\n\nMarkdown content here...",
    "excerpt": "A guide to energy risk fundamentals",
    "category": "Risk Management",
    "tags": "risk, energy, fundamentals",
    "cover_image": "https://example.com/image.jpg"
}
```

Validation:
- Title and content are required
- Content must be at least 100 characters
- Slug is auto-generated from the title (with collision handling via random suffix)
- If no excerpt is provided, it's auto-generated from the first 200 characters of content

**Add comment request body (authenticated):**
```json
{
    "content": "Great article, thanks for sharing!"
}
```

**Add comment request body (guest):**
```json
{
    "content": "Very insightful analysis.",
    "guest_name": "John"
}
```

Comment validation:
- Content required, max 2000 characters
- Guest name required if not authenticated

### Admin API (requires `X-Admin-Token` header)

| Route | Method | Description |
|-------|--------|-------------|
| `GET /api/blog/admin/stats` | GET | Blog statistics (published/pending/rejected/draft/total counts) |
| `GET /api/blog/admin/posts` | GET | List all posts (optional `?status=` filter) |
| `GET /api/blog/admin/posts/{id}` | GET | Get single post detail |
| `POST /api/blog/admin/posts` | POST | Create and immediately publish an article |
| `PUT /api/blog/admin/posts/{id}/status` | PUT | Update post status (publish/reject/draft) |
| `DELETE /api/blog/admin/posts/{id}` | DELETE | Permanently delete a post and its comments |

**Admin create post:** Same body as user create, but status is set to `published` immediately and `author_name` is "EnergyRiskIQ", `author_type` is "admin".

**Update status request body:**
```json
{
    "status": "published",
    "rejection_reason": ""
}
```

Valid status values: `published`, `rejected`, `pending`, `draft`.

## Security

### Password Hashing
Passwords are hashed using **bcrypt** with automatic salt generation. The `_hash_password()` function uses `bcrypt.hashpw()` with `bcrypt.gensalt()`. Password verification uses `bcrypt.checkpw()` for constant-time comparison.

### Session Management
Sessions are stored in the `blog_sessions` database table (not in-memory). This means:
- Sessions survive server restarts and deployments
- Sessions work correctly across multiple worker processes
- Expired sessions are filtered out at query time (`WHERE expires_at > NOW()`)
- Sessions are 30-day duration with a 64-character hex token

### Cookie Security
The `blog_token` cookie is set with:
- `httponly=True` — not accessible via JavaScript
- `secure=True` — only sent over HTTPS
- `samesite='lax'` — CSRF protection
- `max_age=30*86400` — 30-day browser-side expiry

### Content Security
- All user-generated text is HTML-escaped before rendering using `_esc()` (textContent-based escaping)
- Markdown rendering applies escaping before transformation, preventing XSS through article content
- Link URLs in Markdown are sanitized to only allow `http://`, `https://`, and `/` (relative) schemes — `javascript:` and `data:` URLs are stripped
- Admin API error messages never expose internal exception details

### User Isolation
Blog users are completely separate from EnergyRiskIQ platform users:
- Different database tables (`blog_users` vs `users`)
- Different auth mechanism (cookie vs X-User-Token header)
- Different session storage (`blog_sessions` vs `sessions`)
- No cross-referencing between the two user systems

## Markdown Rendering

The blog uses a custom lightweight Markdown renderer (`_render_markdown_basic()`) that supports:

- `# Heading 1`, `## Heading 2`, `### Heading 3`
- `**bold**` and `*italic*`
- `` `inline code` ``
- `[link text](url)` — with URL scheme sanitization
- `- list items` and `* list items` — rendered as `<ul>`
- Paragraph separation via double newlines

Content is HTML-escaped before Markdown transformation, preventing XSS injection.

## Content Features

### Slug Generation
Titles are converted to URL-friendly slugs using:
1. Unicode normalization (NFKD) to ASCII
2. Remove non-word characters (except hyphens and spaces)
3. Replace spaces/multiple hyphens with single hyphens
4. Truncate to 200 characters
5. On collision, append a random 6-character hex suffix

### Reading Time
Calculated as `word_count / 200`, rounded up, minimum 1 minute.

### View Counting
Incremented on each article page load. Fire-and-forget (errors silently ignored).

### Auto-Excerpt
If no excerpt is provided when creating a post, the first 200 characters of content are used, broken at the last word boundary with an ellipsis appended.
